# Homepage Accessibility Blueprint

> Accessibility is not a feature. It is a fundamental requirement of thoughtful design.

---

# Purpose

Every visitor should experience Vivaha with equal dignity,

regardless of

Ability.

Technology.

Device.

Environment.

Accessibility is not an enhancement.

It is the baseline.

---

# Philosophy

A welcoming home welcomes everyone.

Vivaha follows the same principle.

No visitor should encounter unnecessary barriers.

---

# Principles

Accessibility should be

Invisible.

Consistent.

Complete.

Respectful.

Predictable.

---

# Semantic Structure

Homepage should contain

```
<header>

↓

<nav>

↓

<main>

↓

<section>

↓

<footer>
```

Never skip semantic hierarchy.

---

# Heading Structure

One

H1

↓

Section

H2

↓

Subsection

H3

Never skip heading levels.

---

# Keyboard Navigation

Every interactive element

Reachable.

Visible.

Predictable.

Logical.

Support

Tab.

Shift + Tab.

Enter.

Escape.

Arrow Keys where appropriate.

---

# Focus States

Every focus state must remain visible.

Never remove browser focus without replacement.

Focus should remain

High contrast.

Easy to locate.

Consistent.

---

# Colour Contrast

All text

Minimum WCAG AA.

Interactive components

AA minimum.

AAA preferred where practical.

Never communicate meaning using colour alone.

---

# Typography

Readable sizes.

Comfortable spacing.

Generous line height.

Maximum

75 characters per line.

Avoid decorative fonts.

---

# Images

Every image requires

Meaningful alt text.

Decorative imagery

Empty alt.

Never place important text inside images.

---

# Icons

Icons require

Accessible labels.

Tooltips are not labels.

Icons should never communicate meaning alone.

---

# Forms

Every field contains

Visible label.

Helper text.

Accessible error.

Autocomplete.

Correct input mode.

Never rely on placeholders.

---

# Motion

Respect

prefers-reduced-motion.

Essential feedback remains.

Decorative motion disappears.

---

# Screen Readers

Support

Landmarks.

Regions.

Buttons.

Forms.

Navigation.

Dialogs.

Status announcements.

Dynamic content updates.

---

# Responsive Accessibility

Accessibility remains identical across

Desktop.

Tablet.

Mobile.

Touch devices.

Keyboard users.

Screen readers.

---

# Error Prevention

Explain errors clearly.

Preserve entered information.

Offer recovery.

Never blame users.

---

# Touch Targets

Minimum

44×44px.

No exceptions.

---

# Language

Homepage should declare

Document language.

Readable wording.

Plain language.

Avoid jargon.

---

# Performance

Accessibility should never reduce performance.

Prefer

Semantic HTML.

Native elements.

Minimal ARIA.

Avoid unnecessary wrappers.

---

# Testing Checklist

Keyboard only.

Screen Reader.

Reduced Motion.

Zoom 200%.

Mobile.

Tablet.

Desktop.

High Contrast.

Dark Mode.

Light Mode.

---

# AI Implementation Rules

Always

Use semantic HTML.

Maintain heading hierarchy.

Support keyboard navigation.

Support screen readers.

Use descriptive labels.

Never

Hide focus.

Use inaccessible colours.

Replace labels with placeholders.

Ignore reduced motion.

---

# Review Checklist

□ Can the homepage be used entirely by keyboard?

□ Are all headings semantic?

□ Are all images described?

□ Are forms fully labelled?

□ Is reduced motion supported?

□ Does colour meet contrast requirements?

□ Is the homepage understandable without a mouse?

Only if every answer is "Yes" should accessibility be approved.
